# Как работать с Git и GitHub через агента — инструкция для новичка

Эта инструкция для человека, который **не программист** и никогда не работал с git.
Хорошая новость: вам не нужно учить команды. Всю техническую работу делает агент
(Claude Code) — вы просто говорите ему по-русски, чего хотите. Инструкция объясняет,
**что** просить и **какими словами**, и даёт минимум понятий, чтобы понимать ответы агента.

Все примеры — на нашем проекте: **https://github.com/poznik/KTPN-CONFIG-N**

---

## 1. Что такое Git и GitHub — на пальцах

**Git** — это «машина времени» для папки с файлами. Она запоминает каждую сохранённую
версию проекта: кто, когда и что изменил. Можно в любой момент посмотреть историю
или вернуть любой файл к прошлому состоянию. Поэтому испортить что-то безвозвратно
почти невозможно — это главное, что нужно знать, чтобы не бояться.

**GitHub** — это сайт (облако), где хранится **общий** экземпляр проекта. Каждый участник
скачивает себе копию, работает у себя на компьютере, а потом отправляет свои изменения
обратно в облако — и остальные их получают.

**Репозиторий** — просто «проект в git»: папка с файлами плюс вся история её изменений.
Наш репозиторий называется `KTPN-CONFIG-N`, его страница в интернете —
https://github.com/poznik/KTPN-CONFIG-N (можно открыть в браузере и посмотреть файлы и историю).

## 2. Пять слов, которые нужно знать

Агент и сообщения об ошибках используют эти термины, поэтому полезно их узнавать:

| Термин | По-русски | Аналогия |
|---|---|---|
| **clone** (клонировать) | скачать проект с GitHub себе на компьютер **в первый раз** | скачать всю папку из облака |
| **pull** (подтянуть) | обновить свою копию до самой свежей версии с GitHub | «синхронизировать» перед работой |
| **commit** (коммит) | точка сохранения: зафиксированный набор изменений **с подписью**, что и зачем сделано | сохранение в игре с комментарием |
| **push** (запушить) | отправить свои сохранённые изменения на GitHub | выложить в облако для всех |
| **conflict** (конфликт) | вы и кто-то другой изменили **одно и то же место** в файле — git просит человека решить, чей вариант оставить | два редактора правили один абзац |

Весь рабочий цикл: **pull → правите файлы → commit → push**. Всё.

## 3. Что нужно настроить один раз

1. **Аккаунт на GitHub** — зарегистрируйтесь на https://github.com (это бесплатно).
2. **Доступ к репозиторию** — попросите владельца проекта добавить ваш аккаунт
   (на GitHub это делается в Settings → Collaborators). Без этого вы сможете скачать
   проект, но не сможете отправлять изменения.
3. **Установленные Git и Claude Code** на вашем компьютере. Если не уверены,
   спросите агента:

   ```text
   Проверь, установлен ли у меня git и настроен ли доступ к GitHub. Если чего-то не хватает — объясни по шагам, что сделать.
   ```

4. **Вход в GitHub с компьютера.** При первой отправке изменений Windows откроет окно
   браузера с входом в GitHub — просто войдите под своим аккаунтом, это запомнится.
   Если агент говорит, что нет авторизации, он подскажет команду (обычно `gh auth login`) —
   выполняется она один раз, вход происходит через браузер.

> ⚠️ **Важно:** пароли и токены никогда не вставляйте в чат с агентом и не сохраняйте
> в файлах проекта. Вход в GitHub всегда происходит через окно браузера.

## 4. Рабочий цикл — как выглядит обычный день

1. Открыли проект → попросили агента **подтянуть свежее** (pull).
2. Поправили файлы (сами или с помощью агента).
3. Попросили агента **показать, что изменилось** — проверили глазами.
4. Попросили **сохранить и отправить на GitHub** (commit + push).
5. При желании открыли https://github.com/poznik/KTPN-CONFIG-N в браузере
   и убедились, что изменения появились (вкладка «Commits»).

## 5. Готовые промпты — копируйте и вставляйте

### 5.1. Скачать проект в первый раз (клонировать)

```text
Склонируй репозиторий https://github.com/poznik/KTPN-CONFIG-N в папку C:\Users\<ВашеИмя>\projects и скажи, когда будет готово.
```

Делается **один раз**. После этого проект живёт у вас в
`C:\Users\<ВашеИмя>\projects\KTPN-CONFIG-N` — дальше работаете в этой папке.

### 5.2. Перед началом работы — обновиться

```text
Подтяни последние изменения этого проекта с GitHub и коротко расскажи, что нового.
```

Делайте это **каждый раз перед началом работы**, особенно если над проектом работает
кто-то ещё. Так вы не будете править устаревшие файлы.

### 5.3. Посмотреть, что вы изменили

```text
Покажи, какие файлы я изменил и что именно в них поменялось. Объясни простыми словами.
```

Полезно перед отправкой: убедиться, что уходит именно то, что вы хотели,
и не затесалось ничего лишнего.

### 5.4. Сохранить и отправить на GitHub (самое главное)

```text
Сохрани мои изменения в git с понятным описанием на русском и отправь на GitHub. Перед отправкой покажи мне список изменений и текст описания — жди моего подтверждения.
```

Агент сам сделает commit и push. Фраза «жди моего подтверждения» — ваша страховка:
вы всегда видите, что именно уходит в общий проект.

### 5.5. Отменить случайные правки

Если что-то испортили и хотите вернуть как было (изменения ещё **не отправляли**):

```text
Я случайно испортил файл <имя файла> — верни его к последней сохранённой версии. Остальные мои изменения не трогай.
```

Если нужно отменить вообще все несохранённые правки:

```text
Отмени все мои несохранённые изменения в проекте и верни его к последней версии с GitHub. Сначала покажи, что именно будет отменено, и жди подтверждения.
```

### 5.6. Посмотреть историю проекта

```text
Покажи последние 10 изменений проекта: кто, когда и что менял. Объясни простыми словами.
```

### 5.7. Если не помните, на чём остановились

```text
Расскажи текущее состояние проекта: есть ли у меня несохранённые изменения, всё ли отправлено на GitHub, не отстаю ли я от общей версии.
```

## 6. Если что-то пошло не так

**Агент говорит про «конфликт» (conflict).**
Это не поломка: вы и коллега изменили одно и то же место, и git спрашивает, чей вариант
оставить. Скажите агенту:

```text
Возник конфликт. Покажи по каждому спорному месту оба варианта, объясни разницу простыми словами и спрашивай меня, какой оставить.
```

**Ошибка «permission denied» или «403» при отправке.**
У вашего аккаунта GitHub нет прав на запись в этот репозиторий. Попросите владельца
добавить вас в Collaborators (см. раздел 3, пункт 2).

**Агент просит авторизоваться в GitHub.**
Нормальная ситуация при первой отправке. Выполните вход через браузер
(см. раздел 3, пункт 4). Пароль агенту не сообщайте.

**Кажется, что всё сломалось.**
Не паникуйте и ничего не удаляйте вручную. Git хранит всю историю — почти любую
ситуацию можно откатить. Просто опишите агенту своими словами, что вы делали
и что видите на экране:

```text
Кажется, я что-то сломал в git. Вот что я делал: <опишите>. Разберись, что произошло, объясни простыми словами и предложи самый безопасный способ всё починить. Ничего не удаляй без моего подтверждения.
```

## 7. Золотые правила

1. **Перед работой — pull, после работы — push.** Начали день — обновились,
   закончили правку — отправили. Тогда конфликты будут редкостью.
2. **Смотрите, что отправляете.** Всегда просите агента показать изменения
   перед отправкой (промпт 5.4 уже так составлен).
3. **Отправляйте небольшими порциями с понятными описаниями.** Одно осмысленное
   изменение — один коммит: «Исправил цены трансформаторов в каталоге» лучше,
   чем «правки» через две недели работы.
4. **Никаких паролей и секретов в файлах проекта.** Всё, что попало на GitHub,
   видят все участники, а удалить из истории по-настоящему сложно.
5. **Не бойтесь спрашивать агента.** «Объясни, что это значит», «а это безопасно?»,
   «покажи, что изменится, прежде чем делать» — совершенно нормальные просьбы.
