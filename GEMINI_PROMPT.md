# Промт для Gemini: проверка проекта KTPN Configurator

Ты работаешь как второй инженер-ревьюер проекта `KtpnConfigurator_Final`. Не трогай секреты, API-ключи, `.env`, `.continue`, `.codex` и пользовательские настройки IDE. Твоя задача - дать практичный технический разбор без переписывания архитектуры ради переписывания.

## Контекст

Проект - WPF/.NET приложение для конфигурации КТПН:

- `src/KtpnConfigurator.App` - WPF интерфейс, MVVM, экспорт, хранение проектов.
- `src/KtpnConfigurator.Core` - каталог, расчет, валидация, генерация документов.
- `tests/KtpnConfigurator.Tests` - xUnit golden tests.
- `build.ps1` - публикация готового приложения в `build/KtpnConfigurator_V6`.

Последние изменения:

- исправлен WPF startup crash;
- экспорт Excel/PDF заблокирован при ошибках проектирования;
- интерфейс переведен на нейтральную рабочую палитру без ярких зеленых/розовых/синих акцентов;
- добавлены smoke/self-test режимы для проверки запуска.

## Что проверить

1. Проверь `src/KtpnConfigurator.App/MainWindow.xaml`, `App.xaml` и `ViewModels/MainViewModel.cs`.
   Найди возможные ошибки XAML, несуществующие MaterialDesign ресурсы, плохие контрасты, слишком яркие цвета, визуально спорные места и риск обрезания текста.

2. Проверь, что статус проекта корректно отображает:
   - ошибки проектирования;
   - предупреждения;
   - успешное прохождение проверок.
   Цвета должны быть спокойными, но различимыми.

3. Проверь расчетную часть:
   - `CalculationEngine`;
   - `ValidationEngine`;
   - модели `ProjectConfig`, `CalculationResult`, `Catalog`.
   Ищи реальные дефекты: неверные граничные условия, риск `null`, несогласованные единицы измерения, ошибки при ручных переопределениях.

4. Проверь генерацию документов:
   - производственный приказ;
   - паспорт металлоконструкции;
   - чек-лист ОТК;
   - экспорт Excel/PDF.
   Особенно проверь, не формируются ли документы при критических ошибках валидации.

5. Проверь данные каталогов и шаблонов.
   Укажи, где приложение может падать или молча давать неверный результат при поврежденных/неполных JSON-файлах.

6. Проверь `build.ps1`.
   Убедись, что публикация создает рабочую папку `build/KtpnConfigurator_V6`, не зависит от случайного текущего каталога и не ломает запуск `.exe`.

7. Запусти или предложи точные команды проверки:
   - `dotnet build KtpnConfigurator.slnx -c Release --no-restore`
   - `dotnet test KtpnConfigurator.slnx -c Release --no-build`
   - `dotnet run --project src/KtpnConfigurator.App/KtpnConfigurator.App.csproj -c Release --no-build -- --smoketest-ui`
   - `powershell -ExecutionPolicy Bypass -File ./build.ps1`

## Формат ответа

Сначала дай список найденных проблем, отсортированный по важности:

- `Critical` - приложение падает, теряет данные или выдает опасно неверный результат.
- `High` - ломает рабочий сценарий пользователя.
- `Medium` - заметная ошибка, но есть обходной путь.
- `Low` - косметика, читаемость, поддерживаемость.

Для каждой проблемы укажи:

- файл и строку;
- что именно не так;
- как воспроизвести или почему это риск;
- конкретное исправление.

Если проблем нет, прямо напиши: `Критичных проблем не найдено`, затем перечисли остаточные риски и какие проверки были выполнены.

Не предлагай общие советы. Нужны только реальные задачи, которые можно сразу отдать в работу.
