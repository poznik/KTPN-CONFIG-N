# Полный аудит KtpnConfigurator

Дата: 2026-06-14

## Уточнение 2026-06-19

Текущий приоритет изменен: схемы пока не развиваем и не трогаем. Все рекомендации этого аудита по архитектуре схем, визуальным golden-test, листам схем, УГО, DXF/SVG и экспорту схем считать отложенными до отдельного решения.

Сейчас нужно довести до конца остальное:

- корпус и паспортные параметры КТПН: климат, IP, заземление, двери, замки, фиксаторы;
- РУВН и РУНН без углубления в сложную селективность;
- отходящие линии: назначение, имя, резерв, учет, ТТ;
- собственные нужды: мощность, ток, обогрев, вентиляция, освещение, розетка, РИСЭ;
- база оборудования и статусы достоверности;
- проверки проекта с привязкой к вкладкам;
- документы, спецификация, чек-листы и версия сборки.

Если схема начнет мешать выпуску остального функционала, ее можно будет временно отключить отдельной задачей. До такого решения код схем не менять.

## 1. Краткий вывод

Проект уже перешел из стадии "форма вместо Excel" в стадию инженерного конфигуратора: есть WPF-приложение, Core без зависимости от UI, JSON-каталоги, расчетное ядро, проверки, документы, экспорт PDF/Excel, схема, сохранение проектов и тесты.

Главная проблема теперь не в отсутствии отдельных функций, а в архитектуре роста. Новые возможности добавляются в те же крупные файлы и в ручную геометрию схемы. Из-за этого приложение становится трудно расширять, а схема долго не будет похожа на готовый рабочий проект, если продолжать только двигать линии и подписи.

Нужен следующий уровень:

- перейти от ручной картинки схемы к инженерной модели схемы;
- разделить один большой UI и одну большую ViewModel на модули;
- сделать каталоги строгими, версионированными и проверяемыми;
- сделать спецификацию единым источником состава оборудования;
- добавить версию проекта и миграции для старых файлов;
- разделить чистую схему в приложении и рабочие листы для выпуска.

## 5. Данные и каталоги

Текущие объемы:

- `transformers.json` - 73 трансформатора.
- `apparatus.json` - 18 старых записей аппаратов.
- `device_models.json` - 21 модель устройства.
- `diagram_symbols.json` - 20 символов.
- `diagram_rules.json` - правила листов, обозначений и стиля.
- `doc_templates.json` - шаблоны приказа и чек-листа.

Проблемы:

1. `device_models.json` и `apparatus.json` частично дублируют смысл. Дальше нужно выбрать один основной каталог устройств, а старый `apparatus.json` оставить как совместимость или перенести в новую структуру.
2. В `device_models.json` только 2 записи имеют `sourceConfidence = verified`, остальные 19 требуют проверки.
3. В `diagram_symbols.json` 7 записей `verified`, 13 требуют проверки.
4. Нет JSON Schema, поэтому опечатка в имени поля или пустой обязательный параметр может пройти незамеченной.
5. У устройств нет стабильного `id`. Сейчас связи идут по строкам `Type`, `Manufacturer`, `Model`. Это хрупко для переименований.
6. В справочниках мало технических параметров для реального подбора: отключающая способность, исполнение, габарит аппарата, тип установки, количество модулей, класс точности ТТ, нагрузка вторичных цепей, совместимость счетчика и ТТ.

Что нужно добавить:

- `deviceId` для каждой модели.
- `category`, `purpose`, `installationArea`, `voltageClass`, `ratedCurrent`, `breakingCapacity`, `poles`, `dimensions`, `mounting`.
- `source` как объект: документ, ссылка/файл, дата проверки, статус.
- JSON Schema для всех баз.
- Тест, который валидирует все JSON по схемам.
- Диагностическую вкладку "База данных": сколько записей проверено, сколько требует проверки, где пустые поля.

## 6. Модель проекта

Текущий `ProjectConfig` понятный, но плоский. В нем много отдельных полей:

- `PvrOn`, `ReOn`, `AvInOn`;
- `AvOn`, `AvQty`, `RpsOn`, `RpsQty`;
- `HasCt`, `HasMeter`, `CtRatio`;
- список `OutgoingFeeders`;
- `AuxiliaryNeeds`.

Проблемы:

1. Вводные устройства РУНН заданы набором bool-полей. UI позволяет потенциально включить несколько вводных аппаратов одновременно. Для реального проекта это может быть допустимой цепочкой или ошибкой, но это не описано моделью.
2. Отходящая линия содержит только `Number`, `DeviceType`, `Manufacturer`, `Model`, `Nominal`, `TtRatio`, `HasMeter`. Не хватает назначения линии, потребителя, кабеля, резерва, типа учета, счетчика, испытательной коробки, клемм, комментария.
3. `Clone()` делает поверхностную копию через `MemberwiseClone`. Для сложной модели это риск: вложенные списки и `AuxiliaryNeeds` остаются общими, если вызывающий код не скопировал их вручную.
4. Нет `ProjectVersion`. Пока десериализация держится на значениях по умолчанию, но дальше нужны миграции.
5. Часть значений зависит от русских строк вроде `"Нет"`, `"Тупиковая"`, `"Проходная"`, `"Фотореле"`. Для универсальности лучше хранить код/enum, а отображаемый текст держать отдельно.

Рекомендуемая новая структура:

```csharp
ProjectConfig
  ProjectMeta
  TransformerConfig
  EnclosureConfig
  HvSwitchgearConfig
  LvSwitchgearConfig
    List<InputDeviceConfig>
    MeteringConfig
    List<OutgoingFeederConfig>
  AuxiliaryNeedsConfig
  ManualOverrides
```

Для фидера:

```csharp
OutgoingFeederConfig
  Id
  Number
  Name
  Purpose
  DeviceRef
  Nominal
  Cable
  IsReserve
  MeteringConfig
  Notes
```

## 7. Расчет и проверки

Сейчас `CalculationEngine` хорошо решает задачу переноса Excel:

- длина, ширина, высота;
- масса основания и корпуса;
- масса брутто;
- номинал ввода;
- сечения шин из базы трансформатора.

Что не хватает для универсального конфигуратора:

1. Расчет состава оборудования должен быть отдельным от расчета габаритов.
2. Нужно считать габарит РУНН от числа аппаратов, их типоразмеров и исполнения, а не только от количества АВ/РПС.
3. Для ТТ нужен подбор коэффициента по току линии или трансформатора, а не только выбор из списка.
4. Для отходящих линий нужны ток нагрузки, кабель, назначение, резерв, учет.
5. Проверки должны иметь коды, привязку к полю UI и уровень блокировки экспорта.
6. Нужно явно решить правило вводных устройств: один выбранный аппарат или допустимая последовательность.

Нужен новый слой:

- `EquipmentSelectionEngine` - подбирает и нормализует аппараты.
- `EngineeringValidationEngine` - дает ошибки с кодами и привязкой к полям.
- `LayoutSizingEngine` - считает габариты шкафов/секций от состава.

## 8. UI/UX

Текущий интерфейс рабочий, но перегружен. Главная проблема - не стили, а сценарий.

Проблемы:

1. `MainWindow.xaml` содержит весь интерфейс сразу.
2. `MainViewModel` содержит состояние проекта, каталоги, каскадный выбор, команды, документы, схему и пересчет.
3. Вводные устройства сделаны чекбоксами. Пользователь не понимает, это альтернативы или последовательность.
4. Отходящие линии сделаны через `ItemsControl` и ручную сетку. Для реального проекта удобнее `DataGrid` или мастер фидеров.
5. Ошибки показываются списком в результате, но не рядом с конкретным полем.
6. Документы показываются как plain text. Для проверки удобно, но для выпуска нужен предпросмотр ближе к итоговому PDF/Excel.
7. Схема находится отдельной вкладкой. При активной настройке РУНН хорошо бы иметь маленькое живое превью справа или снизу.

Как улучшить:

- Разделить UI на UserControl: `TransformerView`, `HvInputView`, `LvInputView`, `OutgoingFeedersView`, `AuxiliaryNeedsView`, `ResultsView`, `DiagramView`, `DocumentsView`.
- Разделить ViewModel аналогично.
- Сделать `ProjectShellViewModel`, который координирует пересчет и хранит `ProjectConfig`.
- Для вводного РУНН сделать режим: `ПВР/NH`, `РЕ`, `АВ`, `Составной ввод`.
- Для отходящих линий сделать таблицу с командами: добавить, копировать, удалить, переместить, включить учет.
- В строке фидера показывать только основные поля, а учет/ТТ/кабель - в раскрытии.
- Добавить панель "Проблемы проекта" с переходом к полю.
- Добавить режимы сложности: "Базовый" и "Расширенный".

## 9. Схемы - отложено

Этот раздел оставить как архив идей. Сейчас схемы не трогать.

Текущая схема стала чище: без рамки, с шинами, вводом, ТТ, отходящими линиями, ЩСН. Но она все еще выглядит как программная иллюстрация, а не как комплект проектной документации.

Причина: схема сейчас строится ручным кодом в `SingleLineDiagramSceneBuilder`. Это хороший переходный вариант, но не финальная архитектура.

Основные проблемы:

1. Нет отдельной доменной модели схемы. Конфигурация сразу превращается в координаты и примитивы.
2. Нет универсального layout-движка. Координаты фиксированы, а элементы добавляются условиями.
3. `diagram_symbols.json` содержит `symbolGeometry`, `letterCode`, `designationPattern`, но геометрия почти полностью в C#.
4. Нет нескольких листов: однолинейная схема, цепи учета, собственные нужды, ведомость элементов.
5. Нет визуального сравнения с эталоном `2206.pdf`.
6. Нет SVG/DXF/DWG-экспорта. PDF/Excel получают PNG-картинку.
7. Цепи учета смешаны с общей однолинейной схемой.
8. ТТ на фидерах компактнее, чем раньше, но нужно доработать стандартный вид и отдельный подробный лист учета.

Нужный подход:

```text
ProjectConfig
  -> DiagramModelBuilder
  -> DiagramGraph / DiagramSheetSet
  -> DiagramLayoutEngine
  -> CadScene
  -> Renderer: WPF, PNG, SVG, PDF
```

Ключевые классы:

- `DiagramNode` - аппарат, шина, ТТ, счетчик, шкаф, клеммник.
- `DiagramConnection` - силовая связь, N/PE, цепь учета, управление.
- `DiagramSheet` - лист с типом: oneLine, metering, auxiliary, list.
- `DiagramLayoutEngine` - считает позиции и зоны подписей.
- `DiagramSymbolLibrary` - рисует УГО по `symbolGeometry`.
- `DiagramStyle` - толщины линий, шрифт, отступы, режим чистой схемы или листа ЕСКД.

Нужны два режима отображения:

1. `CleanPreview` - чистая схема без рамки, как сейчас просит пользователь для вкладки.
2. `WorkingSheet` - рабочий лист с рамкой, основной надписью и комплектом листов для выпуска.

Архивные идеи по схемам, не выполнять сейчас:

1. Создать `DiagramModelBuilder`, который строит список узлов и связей без координат.
2. Перевести текущую однолинейную схему на этот model-first подход.
3. Вынести все символы из `SingleLineDiagramSceneBuilder` в `DiagramSymbolLibrary`.
4. Добавить отдельный лист `Цепи учета`: TA1-TA3, PI1, клеммы, испытательная коробка, связи.
5. Визуальный golden-test схемы - только как отложенная идея.
6. Использовать `reference_materials\prim\Схемы\2206.pdf` как эталон стиля, но хранить путь к эталонам явно в документации или тестовых ресурсах.

## 10. Документы и спецификация

Сейчас документы генерируются в `Documents.cs`. Файл делает слишком много:

- шаблоны;
- plain text;
- приказ;
- паспорт;
- чек-лист;
- спецификацию;
- форматирование строк;
- поиск источника данных.

Проблемы:

1. Есть отдельный `SpecificationBuilder`, но реальная богатая спецификация собирается в `DocumentBuilder.BuildSpecification`. Получается дублирование.
2. Спецификация должна быть доменной моделью, а документы должны только отображать ее.
3. Приказ, паспорт и чек-лист лучше строить отдельными builder-классами.
4. Документы пока не имеют полноценного шаблонизатора с полями, условиями и таблицами.
5. PDF/Excel-экспорт форматирует документы заново, а не использует единый layout документа.

Рекомендуемая структура:

- `ProductionOrderBuilder`
- `PassportBuilder`
- `ChecklistBuilder`
- `SpecificationBuilder`
- `DocumentTemplateEngine`
- `DocumentExportModel`

`SpecificationBuilder` должен стать единым источником состава оборудования. Все строки спецификации должны иметь:

- позицию;
- обозначение;
- наименование;
- тип/модель;
- производитель;
- номинал/характеристику;
- количество;
- единицу;
- источник данных;
- статус проверки.

## 14. Приоритет на ближайшую работу

Самый правильный порядок:

1. Зафиксировать текущее состояние в git.
2. Добавить `ProjectVersion` и глубокий `Clone`.
3. Нормализовать вводные устройства РУНН.
4. Перевести спецификацию в единый `SpecificationBuilder`.
5. Создать `DiagramModelBuilder` без координат.
6. Перенести текущую чистую схему на новую модель.
7. Вынести символы в `DiagramSymbolLibrary`.
8. Добавить отдельный лист цепей учета.
9. Разделить UI фидеров и собственных нужд на отдельные View/ViewModel.
10. Добавить JSON Schema и диагностику базы.

## 15. Итог

Приложение можно развивать дальше на текущем стеке C#/WPF. Переписывать все с нуля не нужно. Но схему и UI нужно перестраивать архитектурно: сначала модель, потом layout, потом отрисовка. Тогда схема перестанет быть набором линий и начнет быть инженерным документом, а приложение станет удобным и универсальным, а не набором отдельных исправлений.
