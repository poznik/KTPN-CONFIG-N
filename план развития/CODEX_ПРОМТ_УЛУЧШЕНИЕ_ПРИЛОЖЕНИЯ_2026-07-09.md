# Промт для Codex: улучшение KtpnConfigurator

Дата: 2026-07-09

Этот файл читать перед продолжением разработки приложения. Он нужен как короткая рабочая инструкция: что улучшать, что не трогать, какие решения уже приняты и в каком порядке двигаться.

## Главная задача

Развивать `KtpnConfigurator` как инженерный конфигуратор изделий:

- КТПН;
- 2КТПН;
- НКУ;
- ЩО;
- ВРУ;
- КСО;
- КРУ.

Активная разработка ведется в:

`versions/KtpnConfigurator_Optimized`

Текущую основную версию в корне не ломать. Схемы пока не развивать и не включать, если пользователь прямо не скажет вернуться к схемам.

## Стиль работы

1. Сначала читать текущий код и существующие паттерны.
2. Не делать больших теоретических уходов, если пользователь просит практическую правку.
3. После каждой серьезной правки запускать тесты и по возможности `--smoketest-ui`.
4. После этапа обновлять журнал восстановления или отдельный отчет в `план развития`.
5. В чат отвечать коротко: что сделано, что проверено, где лежит результат.
6. Не возвращать упоминания Gemini/Джимини и посторонние старые переписки.

## Неподвижные решения пользователя

- Схемы пока не трогать.
- Не углубляться в полноценную селективность.
- Текущее простое правило: номинал отходящего автомата не должен быть выше ограничивающего вводного аппарата.
- ТТ подбирать по номиналу соответствующего вводного/ограничивающего автомата.
- Бренд в отходящей линии нужен, но не как общий выбор сверху, а в конкретной строке линии.
- Если назначение отходящей линии РУНН не выбрано, предупреждение не нужно.
- Если ошибок и предупреждений нет, не показывать шумные надписи `0 ошибок / 0 предупреждений`.
- Заказчик выбирается сверху, профиль молча подстраивает проект. Не писать в документах постоянно "по ТЗ такого-то заказчика".
- В документах не должно быть фамилий.
- В чек-листе: книжный A4, без комментариев и без заливки.
- При NH рядом показывать простую надпись `РИСЭ`, если РИСЭ не включена.

## Приоритет 1: довести новые изделия до полезного уровня

### 2КТПН

Сделать 2КТПН не просто вариантом КТПН, а нормальной двухтрансформаторной конфигурацией:

- независимый выбор Т1 и Т2;
- две секции РУНН;
- ввод 1, ввод 2, секционный аппарат;
- привязка каждой отходящей линии к секции 1 или 2;
- проверка: отходящий аппарат не выше своего секционного/вводного ограничения;
- расчет шин по каждой секции;
- расчет массы с учетом второго трансформатора, второго РУВН/РУНН, секционного аппарата;
- отдельные строки в приказе, паспорте и спецификации для Т1/Т2 и секций;
- понятные предупреждения по АВР, параллельной работе и аварийному питанию одной секцией.

### НКУ, ЩО, ВРУ

Сделать линейку панелей реально рабочей:

- активное редактирование каждой панели;
- тип панели: вводная, секционная, линейная, учет, АВР, ЩСН, резерв;
- секция панели;
- аппарат, производитель, модель, номинал;
- Icw/Ipk или хотя бы ток КЗ как поле с проверкой;
- ширина, масса, степень защиты;
- типовые наборы для ЩО и ВРУ;
- итоговая длина ряда, масса, количество панелей;
- документы должны выводить линейку панелей отдельной таблицей.

### КСО, КРУ

Довести линейку ячеек:

- ячейка ввода, отходящая, ТН, секционная, учет, резерв;
- основной аппарат: ВНА/ВНР/РВЗ/вакуумный выключатель;
- РЗА для вакуумного выключателя;
- ТТ/ТН, класс точности, коэффициенты;
- видимые разрывы до и после вакуумного выключателя, если требуется;
- ширина, масса, номинальный ток, ток отключения;
- документы должны выводить линейку ячеек отдельной таблицей.

## Приоритет 2: база оборудования

Нужно постепенно превратить базу оборудования в источник, которому можно доверять:

- разделить оборудование по месту применения: ввод, отходящая линия, ЩСН, РУВН, РУНН, НКУ, КСО, КРУ;
- у каждой позиции хранить статус доверия: проверено, требуется проверка, неизвестно;
- не выдумывать характеристики;
- неподтвержденные параметры показывать как `Требуется проверка`;
- добавить больше реальных серий аппаратов, ТТ, ОПН, РЗА, OPS/ОПС;
- для ОПН РУВН выбирать по пропускной способности, а не по разрядному току;
- для ТШП оставить возможность выбора `50/5` и класса точности.

## Приоритет 3: расчеты

Проверить и улучшить расчеты:

- масса должна быть прозрачной: корпус, трансформатор, аппараты, шины, двери, панели, ячейки, запас;
- шины РУВН, РУНН, N, PE/PEN считать по выбранному ТМГ и материалу;
- для 2КТПН считать шины отдельно по Т1/Т2 и секциям;
- для НКУ/ЩО/ВРУ считать габариты по сумме панелей;
- для КСО/КРУ считать габариты по сумме ячеек;
- добавить проверку явных противоречий: номинал аппарата, секция, ток, материал шин, IP, климат, двери.

## Приоритет 4: вкладки и интерфейс

Улучшать интерфейс не декоративно, а для работы:

- меньше визуального шума;
- крупнее и ярче статусы: `Заполнено`, `Требуется проверка`, `Есть ошибка`;
- скрывать нулевые счетчики ошибок;
- делать активные строки вместо таблиц, которые выглядят заблокированными;
- в каждом изделии показывать только нужные вкладки;
- для КТПН/2КТПН оставить РУВН, РУНН, трансформатор и корпус;
- для НКУ/ЩО/ВРУ показывать панельную компоновку;
- для КСО/КРУ показывать ячейки среднего напряжения;
- не выводить лишние объясняющие тексты внутри приложения.

## Приоритет 5: документы

Проверять четыре главных документа:

- приказ;
- паспорт металлоконструкций;
- чек-лист;
- спецификация.

Что улучшать:

- документы должны соответствовать выбранному изделию;
- не писать лишний текст про заказчика и его ТЗ;
- не дублировать одно и то же в разных местах;
- в приказе раздел `Параметры изделия` без лишних подписей напротив каждого параметра;
- спецификация должна явно показывать состав изделия;
- для новых изделий нужны отдельные таблицы: трансформаторы, панели, ячейки, аппараты, шины, двери, ОПС/ЩСН;
- фамилии не возвращать.

## Приоритет 6: профили заказчиков

Профили заказчиков должны работать тихо:

- пользователь выбирает заказчика в шапке;
- приложение подставляет RAL, IP, климат, двери, замки, маркировку, логотип, ОПС, требования корпуса;
- в документах не объяснять, что это "по ТЗ заказчика";
- если профиль не может применить что-то автоматически, показать только короткое предупреждение в проверках.

## Что добавить позже

- библиотеку типовых компоновок по изделиям;
- мастер "создать изделие из шаблона";
- расчет КЗ и проверку Icw/Ipk;
- нормальные шаблоны РЗА для КРУ/КСО;
- разные профили документов под КТПН, 2КТПН, НКУ, ЩО, ВРУ, КСО, КРУ;
- экспорт ведомости покупных изделий;
- журнал ручных отклонений: где пользователь согласился с ошибками и почему проект можно отдать в работу.

## Команды проверки

Из папки:

`versions/KtpnConfigurator_Optimized`

запускать:

```powershell
dotnet test KtpnConfigurator.slnx -c Release
powershell -ExecutionPolicy Bypass -File .\build.ps1
.\build\KtpnConfigurator_Optimized\Конфигуратор КТПН Optimized.exe --selftest
.\build\KtpnConfigurator_Optimized\Конфигуратор КТПН Optimized.exe --smoketest-ui
```

Если приложение открыто, перед публикацией закрыть только процесс Optimized exe, не трогая другие версии.
