# Аудит используемой базы оборудования

Дата: 2026-06-19

## 1. Какая база реально используется приложением

Приложение загружает справочники из `src/KtpnConfigurator.App/Data` через `CatalogStore.Load()`:

- `transformers.json` - трансформаторы, габариты, массы, ток НН, шины;
- `apparatus.json` - старый укрупненный каталог типов аппаратов и диапазонов производителей;
- `options.json` - списки UI: производители, номиналы, напряжения, исполнения;
- `device_models.json` - новый рабочий каталог моделей оборудования для UI, схемы и спецификации;
- `diagram_symbols.json` - УГО и буквенные обозначения;
- `diagram_rules.json` - правила оформления схемы.

Для выбора моделей в UI главным является `device_models.json`. Если модели нет, часть UI откатывается к старому `apparatus.json`, но при этом теряются точные номиналы, источник и будущие параметры защит.

## 2. Что было найдено

JSON-файлы были валидны, но база оборудования была неполной:

- `device_models.json` содержал 21 запись;
- в `options.json` были производители, для которых не было моделей в `device_models.json`;
- для АВ отсутствовали записи по EKF, ДКС/DKC, CHINT и части серий КЭАЗ/IEK/Контактор;
- для ПВР/NH отсутствовали IEK, EKF, CHINT, Апатор-Электро;
- для РЕ отсутствовали IEK, CHINT, Кореневский завод НВА;
- для РПС отсутствовали КЭАЗ и CHINT;
- защитные аппараты не имели отдельного блока метаданных для будущих ПУЭ-проверок: тип защиты, расцепитель, Icu/Ics, категория селективности.

## 3. Что обновлено

Обновлен `device_models.json`:

- количество записей увеличено с 21 до 42;
- добавлены недостающие серии из используемого `apparatus.json`;
- все производители из `options.json` теперь покрыты записями в `device_models.json`;
- для защитных аппаратов добавлены поля:
  - `protectionKind`;
  - `releaseType`;
  - `tripCurve`;
  - `breakingCapacityIcuKa`;
  - `serviceBreakingCapacityIcsKa`;
  - `selectivityCategory`;
  - `protectionNotes`.

Расширен `DeviceModel` в `src/KtpnConfigurator.Core/Models/Catalog.cs`, чтобы приложение могло загружать эти поля.

Добавлены тесты:

- `DeviceModelsCoverConfiguredEquipmentLists` - проверяет, что все производители из UI покрыты моделями;
- `ProtectiveDeviceModelsHaveProtectionMetadata` - проверяет наличие блока защитных метаданных у АВ, РПС, ПВР/NH и РЕ.

## 4. Важное ограничение по достоверности

Новые записи, добавленные из внутреннего `apparatus.json`, помечены как `needsVerification`, если нет подтверждения по конкретному паспорту или каталожному листу производителя.

Поля `breakingCapacityIcuKa`, `serviceBreakingCapacityIcsKa`, `tripCurve`, `releaseType` и `selectivityCategory` намеренно оставлены пустыми/null там, где нет точной привязки к артикулу. Для ПУЭ и селективности нельзя подставлять эти значения "примерно по серии".

## 5. Быстрая внешняя сверка серий

По официальным/производственным источникам точечно подтверждено, что добавленные серии существуют или присутствуют в каталогах/документации:

- КЭАЗ: OptiMat D, OptiMat A, ВА57;
- IEK: ВА88, ВА07;
- EKF: PROxima, AVERES;
- CHINT: NM8N, NXm, NA8G, NH40;
- ДКС/DKC: YON MAX;
- Контактор: Электрон/Электрон Про, ПРОТОН;
- Апатор-Электро: ARS.

Эта сверка не заменяет паспортную проверку конкретного аппарата. Для инженерного выпуска нужно проверить точный артикул, исполнение, номинал, отключающую способность, расцепитель и таблицы селективности.

## 6. Что нужно сделать следующим этапом

1. Разделить номенклатуру и защитные характеристики:
   - `device_models.json` оставить как каталог выбора;
   - добавить `protection_devices.json` для точных защитных параметров по артикулу.

2. Добавить обязательные поля для ПУЭ-проверок:
   - Icu/Ics, кА;
   - тип расцепителя;
   - B/C/D или LSI/LSIG;
   - Ir/Isd/Ii и выдержки времени;
   - Icw для селективных аппаратов;
   - ссылка на паспорт/каталог;
   - дата проверки источника.

3. Добавить `selectivity_tables.json`:
   - пары "вышестоящий аппарат -> нижестоящий аппарат";
   - предельный ток селективности;
   - источник таблицы производителя.

4. В UI добавить статус:
   - "модель есть";
   - "ПУЭ-параметры заполнены";
   - "селективность подтверждена";
   - "данных недостаточно".
